## 1. One-paragraph orientation

**Vela** (live at https://vela.study) is a fine-art figurative editorial platform that combines a curated image library, a magazine, an adaptive image-sequencing engine called *Reincarnation*, and a writing/research workspace called *Loom*. Its register is contemplative, editorially rigorous, desire-and-emotion-oriented. The editorial spine is shifting from "sex + art + religion" toward a broader "whole human" axis covering emotion, developmental psychology, and developmental theology, while keeping the visual core. The codebase is a single Next.js 16 + React 19 + Tailwind 4 app on Vercel, backed by Supabase Postgres, with Anthropic models doing the AI work.

**Loom** is one product *inside* Vela: a long-form, draft-oriented writing workspace anchored to memoir corpus passages tagged on an Experience × Emotion taxonomy. It is currently used for compensated research sessions (Loom is the surface for the *study* — research participation, consent, compensation), and is being explored as a self-directed reflection surface as well. Loom is the primary place where any new "writing feature" will land or be informed.

If you only remember three things:

1. **Vela is the platform; Loom is one workspace inside it.** Many other surfaces also touch writing (the Magazine, Mosaic text library, Narrative Intelligence, the Office initiative). They are *related* but distinct.
2. **The corpus is two-grade.** "Curator-grade" passages (~15 hand-labeled per source, themed/charge-tiered) drive editorial work. "Research-bulk-grade" passages (~50–150 paragraph-respecting chunks per source) drive retrieval and audits. Loom reads the bulk grade.
3. **Adaptation is a first-class system, not a feature.** *Reincarnation* (for images) and the in-flight *Compass* (for written content) are the adaptive-intelligence layers. Anything you build for writing should expect to interoperate with — or eventually be measured by — these.

## 4. Loom — purpose & scope

### 4.1 What Loom is

Loom is a **long-form study workspace** where memoir, image responses, and reflection threads weave together. The labs-page intent paragraph (literal copy from the running app):

> *"A long-form study workspace where memoir, image responses, and reflection threads weave together. Currently used for compensated research sessions; we are exploring whether the same surface should exist for self-directed reflection."*

Practically it is:

- A **draft editor** (markdown body, per-track, per-user, persisted to loom_draft_sessions).
- A **retrieval pane** — pulls memoir-corpus passages relevant to the draft on demand via POST /api/writer/retrieve. Retrieval is **axis-aware**: an axis (an Experience category and/or a primary Emotion) restricts the pool to passages tagged with that axis, intersected with semantic similarity to the draft.
- An **emotion-vocabulary panel** — given a primary emotion, surfaces ~5 representative passages with attribution so the writer can drop them in as quotes (lib/loom/emotion-vocabulary.ts).
- An **emergence log** — captures memory-notes the writer attaches to their draft as they go (loom_emergence_events), optionally pinned to a passage (passage_id + passage_snippet).

### 4.2 Tracks

Loom is organized into **tracks** — slug-named topical containers, each with a default corpus bias. Current tracks (from lib/loom/tracks.ts):

- spiritual-journey — Faith, doubt, practice
- single-dad — Custody, work, tenderness
- gen-x — Cohort texture, irony, fatigue
- social-anxiety — Interior exposure, avoidance
- cult — Belonging, control, exit
- pioneer — Risk, frontier, self-invention
- four-rooms *(provisional)*
- illness-five-years *(provisional)*

Each track currently shares a default corpus filter biased toward research-bulk-chunk memoir passages. Per-track tuning happens as the corpus grows.

### 4.3 The Experience × Emotion taxonomy (load-bearing)

Loom retrieval is built on a **two-axis tag schema** stored in loom_passage_tags (FK to mosaic_passages):

| Field | What |
|---|---|
| experience_category | Slug from a merged top-down + bottom-up taxonomy (lib/loom/taxonomy/experience.ts) |
| experience_subcategory | Optional finer label |
| emotion_primary | One of EMOTION_PRIMARY_OPTIONS (lib/loom/taxonomy/emotion.ts) |
| emotion_secondary | Up to 4 additional primaries |
| valence | −1 (negative) … +1 (positive) |
| arousal | 0 (calm) … 1 (activated) |
| complexity_markers | Subset of COMPLEXITY_MARKERS |
| tag_model, tag_version, tagged_at | Provenance |

**Important:** the **shame family** is intentionally kept as nine *distinct* emotion_primary values (see SHAME_FAMILY_PRIMARIES). Do not collapse them into a single "shame" bucket in any API or feature surface.

**Corpus scope:** memoir bulk = research-bulk-chunk theme ∩ mosaic_sources.source_type = 'book' (lib/loom/taxonomy/memoir-corpus.ts).

### 4.4 Retrieval surface — POST /api/writer/retrieve

The single HTTP entry point for "get me passages relevant to this draft." Body shape (simplified):


json
{
  "draft": "<markdown the writer is currently working on>",
  "axis": {
    "experience_category": "...optional...",
    "emotion_primary": "...optional...",
    "query_text": "...optional override of draft for the semantic side..."
  },
  "corpus_filter": {
    "themes": ["research-bulk-chunk"],
    "passage_ids_allowlist": ["...optional manual id set..."]
  }
}


Behavior:

- If axis is set, results are restricted to loom_passage_tags matching that axis, **intersected** with semantic ranking (when draft or axis.query_text is non-empty).
- If only corpus_filter.passage_ids_allowlist is provided, that's the pool.
- If neither is set, the full corpus filtered by corpus_filter.themes is the pool.

Library entry points: retrieveByAxis, listPassageIdsForAxis in lib/platform/writer-retrieval/cross-axis.ts. Types in lib/platform/writer-retrieval/types.ts.

### 4.5 Loom data model (the tables you'd actually touch)

| Table | Purpose |
|---|---|
| loom_draft_sessions | One row per (user, track, session). Fields: id, user_id, track_slug, title, body_markdown, created_at, updated_at. |
| loom_emergence_events | Memory-notes captured during a session. Fields: id, user_id, track_slug, session_id, memory_note, passage_id, source_book_code, passage_snippet, created_at. |
| loom_passage_tags | Two-axis tags on memoir passages (see §4.3). FK passage_id → mosaic_passages.id. |

## 5. Vela vs Loom — key differences

The clearest way to hold this distinction:

| | **Vela** | **Loom** |
|---|---|---|
| **What it is** | The whole platform: image library + player + magazine + adaptive intelligence + writing workspace + study + membership | One workspace *inside* Vela: long-form drafting against memoir corpus with emotion-axis retrieval |
| **Primary medium** | Images (with surrounding editorial text) | Text (drafts) — with optional retrieved passages |
| **Primary actor** | Reader / participant / member | Writer / study participant in a long-form session |

| **Adaptive engine** | Reincarnation (in production) — scores images, manages pools, builds adaptive queue | None today; Compass (planned) is the equivalent for written content. Loom retrieval is axis-restricted semantic, not adaptive in the Reincarnation sense |

**Where the line blurs (and why it matters for new writing features):**

- The **Magazine** is also "writing." If a feature targets reader experience of finished essays (per-reader variants, impact-rating capture, chain ordering), it lives on the magazine + Compass surface, not Loom.
- The **Mosaic / NI** stack is the *source* layer — the curated text library that feeds both the magazine and Loom retrieval. New ingest, tagging, or theme work usually touches Mosaic/NI, not Loom.
- The **Adaptive Authorship Platform** (lib/platform/) is the cross-property kernel; if a feature is designed to apply to multiple properties (Vela, Bicycle Guides, etc.), it belongs in lib/platform/, not lib/loom/. The lint guard enforces the boundary.
- **Compass** does not exist yet but is funded and named. Anything you design that takes per-article rating + chain-position should expect Compass to ship beneath it.

## 7. Anti-goals + load-bearing rules a writing collaborator should respect

**Rules to honor when proposing or designing a writing feature:**

1. **Don't collapse the shame family.** Nine distinct emotion_primary values are kept on purpose.
2. **Don't bypass the two-grade corpus distinction.** Curator-grade is editorial; research-bulk-grade is retrieval. Conflating them produces confident-but-wrong retrieval (the failure mode is "top hits are unrelated but high-similarity").
3. **Don't default to topic similarity.** Compass's load-bearing concept is *developmental ordering*, not "more like this." If your feature is structurally a recommender, name that — and show how it interoperates with chains.
4. **Don't import Vela-app modules from lib/platform/.** If a feature wants to be cross-property, it belongs in lib/platform/ and must stay clean.
5. **Don't treat Loom as the magazine.** Loom is a drafting surface (writer-side). The magazine is a published surface (reader-side). Features are usually one or the other.
6. **Don't quietly bypass attribution.** Every passage retrieved from Mosaic has an mosaic_sources row with title and author. Surfaced quotes must remain attributable; this is a load-bearing editorial constraint, not just a UX nicety.
7. **Don't treat writing features as pure UI.** The corpus, the tags, the retrieval shape, and the eventual measurement (Compass) are all part of the design surface. The best feature proposals name the data + retrieval shape they need, not just the UI.
8. **Don't moralize about emotion or desire.** Vela treats them as research objects, not problems to solve. Voice register matters; see docs/magazine/VELA-MAGAZINE-VOICE.md if voice is part of the feature.

**Things Vela is deliberately *not* doing right now (so the collaborator can defer noise):**

- Building generic AI writing assistants ("autocomplete this paragraph"). Loom's retrieval-and-attribution shape is intentional and not the same as "ghostwriter."
- Optimizing for SEO / traffic at the expense of editorial register.
- Building cross-cultural content without source-level rigor (the emotion corpus is being expanded *deliberately* with cited primary sources — see docs/research/emotion-corpus-expansion/).
- Replacing human curation. Reincarnation surfaces signal; humans still make the editorial calls.

**Premise (Mike, 2026-05-02):** *"I want to develop a kind of office culture so I don't have to make all decisions on my own. For inspiration, and for fun."*

**The three layers:**

1. **Writer's Desk** (ASN-1092) — 1:1 chat with a single writer (e.g., Dana Weiss). Mike opens a thread; Dana responds *in voice* from her dossier. Mike pastes a draft; Dana reacts. Mike asks "what would you write next week if it were up to you?"; Dana proposes. Streaming Anthropic call (claude-sonnet-4-6); dossier-as-system-prompt. Surface: app/admin/writers/[slug]/desk/.
2. **The Office** (ASN-1093) — multi-writer convening. Mike drops a question or a draft into a meeting; N writers (≤8) respond independently in round 1; in optional round 2, each writer sees the others' responses and *reacts*. Disagreement is the point. Promise.allSettled fan-out. Surface: app/admin/office/.
3. **Writer-initiated pitches** (ASN-1094) — weekly cron fires per active writer; each writer pitches 1–3 of: an article they want to write, a corpus passage they're sitting with, an editorial pushback, or a series proposal. Triage UI (app/admin/office/pitches/) with Pending / Accepted / Rejected / Parked tabs. Accept-as-draft auto-creates a magazine_articles row.

**Schema** (ASN-1091, gates the cluster): writer_threads, writer_messages, office_meetings, office_meeting_responses, writer_pitches. Admin-only RLS in Wave-1.

**Cross-system hooks** (ASN-1095): pitch-→-magazine-draft provenance; "Ask the Office" button on Compass chain/probe authoring surfaces; cost surface (OFFICE_DAILY_USD_LIMIT hard cap, default $5/day); office activity logged to writer_retrieval_usage.

**Pulse dashboard** (ASN-1096): metrics for office activity (pitches per week, acceptance rate, meetings convened, top-engagement writers).

**Load-bearing editorial constraints** (do not soften):

- The Writer's Desk system-prompt frame includes: *"You are not a chatbot — you are a colleague with taste and history. Disagree if you'd disagree. Push back if you'd push back."* This sentence is the difference between Dana sounding like Dana and Dana sounding like Claude with a vocabulary palette.
- The pitch frame includes: *"Pitches you wouldn't bother making aren't pitches."* This pushes writers toward conviction over compliance.
- Round 2 of meetings is the surprising layer — agents often skip "let LLMs see each other"; here it's the whole point. Don't truncate the colleague-context aggressively.
- Writers have **dossiers** (lib/writers/dossier-reader.ts reads from disk). The dossier is the writer's voice. Do NOT layer additional editorial-style instructions on top — if Dana's dossier says she opens with imagery, let her open with imagery. The writers are real to the extent we trust their dossiers.

**Module location:** lib/office/ (kept separate from lib/writers/ which owns the production composition pipeline). The Office is *consumer* of writers, not part of the production surface.

**If your feature work is on this arc:** the external LLM should know about dossiers, the streaming chat shape, the meeting fan-out pattern, the pitch-generation context budget (≤2k tokens of platform state per writer per session), and the editorial register rules above.

### 9.2 Compass (ASN-1061..1087) — the adaptive layer for written content

**One-line:** Reincarnation's sister system *for articles* — per-article rating, calibration probes, chains-as-developmental-orderings.

**Premise (Mike, 2026-05-02):** *"Just like the version of the player where there is the 5-point scale built in at the bottom which enables the user in proceeding to the next image, we should deploy something similar with each article. We will specifically instruct users that this feedback enables us to serve them with content that is a better match for them, over time. In addition to the regular always-on feature we should have wizard and calibration exercises which we request from new users. We go to them and move them up a knowledge chain, as opposed to making them come to us."*

**The load-bearing concept — chains, not topics:**

A **chain** is a *developmental ordering* of articles, not a topic cluster. It has:

- A name (e.g., "developmental theology", "from-I-like-sex-to-whole-human", "leading people: novice → tenured manager")
- A calibration battery (forced-choice + continuum questions that locate a reader on the chain)
- An ordering algorithm (which article to serve next given the reader's current position + what they just read)
- A set of articles tagged into the chain *with positions* (article X is at position 3 in chain A; the same article may be at position 7 in chain B)
- A scoring model (impact ratings + probe responses update the reader's position on the chain)

Topic similarity says *"you read about Augustine, here's another Augustine piece."* Chain ordering says *"you read the entry-level Augustine piece on shame, you rated it as moving you, your position on the developmental-theology chain advanced from waypoint 2 to 4 — here's the waypoint-5 piece."* The reader is **escorted up the chain**, not handed similar items.

**Two measurement primitives:**

1. **Impact rating** (always-on, end of every article): 5-point scale (*how much did this move you?*) + optional direction tag (*more like this* / *shift register* / *less like this*). The shift-register tag is editorially load-bearing: a reader can rate a piece highly *and* signal "I want a different register next." That distinction is invisible to a single-dimension scorer.
2. **Calibration probes** (light, between articles): forced-choice ("this vs. that" — choose which lands more) and continuum placement ("where do you sit on X?" 1–7 ordinal). Probes are how Compass moves from *reactive* (scoring what was read) to *active* (asking the user a sharp question that locates them on a chain in 15 seconds). Reincarnation does not have probes.

**Wave 6 — reader lookup layer:** an inline lookup affordance the reader can hit on any term/concept in an article. Returns a gloss + 2–3 supporting passages from Mosaic. **Lookup events feed the profile** (a reader who looks up "kenosis" three times in a week is positioned differently than one who never does). Optional 0–5 per-lookup impact rating with editorially-load-bearing 0-floor (zero is a real signal — *"I asked, and the answer didn't help"* is worth knowing).

**Schema** lives in lib/compass/ (or packages/factory/src/compass/ per cross-property pattern). Articles can belong to multiple chains; position is per-membership; the schema is a join table (compass_chain_memberships), not a column on articles.

**Cross-property:** Compass is built to apply to Vela *and* Bicycle Guides (sibling property). Anything Compass-shaped should respect the library-core multi-property pattern (property_id columns).

**If your feature work is on this arc:** the LLM should know about the chain shape, the impact-rating + direction-tag pair, the probe primitive (forced-choice and continuum), the per-(user, chain) profile shape, and the lookup-events-feed-profile pattern. Topic-similarity recommenders are the wrong default.

**Vision doc:** docs/VISION-COMPASS.md (load-bearing).

### 9.3 Loom Emotion Surfaces wave (ASN-930..938) + emotion corpus expansion (ASN-1023, ASN-1024, ASN-1025)

### 9.5 Anti-targets — what "writing features" does NOT mean here

The phrase "writing features" frequently maps, in other projects, to one of these — none of which are currently the target:

- **Generic AI writing assistants** ("autocomplete this paragraph", "rewrite this in the style of X"). Vela has voice-rewriting infrastructure for the writer roster; expanding it generically is not a current push.
- **SEO content generators.** Editorial register is the load-bearing axis; SEO is downstream of register, not upstream.
- **Comment/community surfaces.** Vela is reader-facing one-way today (apart from impact ratings, which Compass is building).
- **A "ghostwriter"-style tool that writes essays.** The writer roster + dossiers handle this for the magazine; the human-author surfaces (Loom, Office) are for *human* authoring with retrieval support, not for AI replacing the human.

If a feature proposal drifts into one of these, the LLM has lost the thread.

## 10. How to use this briefing with the external LLM

Suggested first-turn structure when handing off to the external LLM:

1. Paste this entire document as system context (or as the first user turn).
2. State the actual feature ask in your second turn: *"With that context, here's what I want to build…"*
3. Ask the LLM to **restate** Vela vs Loom in its own words before it proposes anything — that catches misreadings cheaply.
4. When it proposes a feature, ask it to name (a) which surface it lives on (Vela / Loom / Magazine / Mosaic / Platform), (b) what corpus shape it needs, (c) what retrieval shape it needs, (d) how it interoperates with Compass and Reincarnation, and (e) what could go wrong if the load-bearing rules in §7 are violated.

That sequence consistently produces grounded proposals from a cold-start collaborator.
